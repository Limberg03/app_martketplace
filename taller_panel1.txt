1. Introducción:


La educación universitaria en el ámbito de las ciencias computacionales y la ingeniería de software es un motor constante de innovación técnica. Semestre tras semestre, los estudiantes desarrollan sistemas complejos, arquitecturas de software robustas y soluciones funcionales como parte de sus exigencias académicas. Sin embargo, existe una brecha sistemática entre el entorno académico y el sector comercial: la inmensa mayoría de estos proyectos, una vez defendidos y calificados, quedan archivados en repositorios locales y su ciclo de vida concluye de manera abrupta. Esta realidad no solo representa una pérdida significativa de capital intelectual, sino que frena el potencial del estudiante para insertarse tempranamente en el mercado tecnológico.


Para transformar este paradigma, el presente proyecto propone el desarrollo de un Marketplace de Aplicaciones de Emprendimiento Digital Académico. El propósito central de esta plataforma es transformar radicalmente el ciclo de vida de un proyecto académico universitario, logrando que deje de ser percibido como una simple tarea para aprobar una materia y se consolide como un activo digital real y 
monetizable
. El sistema servirá como un ecosistema transaccional donde los desarrolladores pueden ofrecer sus sistemas a Pequeñas y Medianas Empresas (Pymes) que demandan soluciones tecnológicas, pero que a menudo no cuentan con el presupuesto para desarrollos a medida desde cero.


La gran novedad y el núcleo innovador de esta propuesta radica en que trasciende la arquitectura tradicional de un simple catálogo web. Mediante la integración profunda de Inteligencia Artificial (IA) y algoritmos de Machine 
Learning
 Analítico, el sistema asume un rol proactivo. Por un lado, incorpora una búsqueda inteligente y semántica impulsada por Procesamiento del Lenguaje Natural (PLN), la cual permite que los compradores busquen software utilizando un lenguaje coloquial. De este modo, la IA interpreta la intención real detrás de la búsqueda y localiza la aplicación que mejor resuelve ese problema específico en la base de datos.


Adicionalmente, el proyecto aborda una de las barreras más críticas para el talento emergente: la falta de experiencia comercial de los estudiantes universitarios. Para mitigar esto, la plataforma actúa como un asesor de ventas mediante la predicción y optimización de precios, analizando características técnicas y tendencias para sugerir el valor óptimo del software, evitando así que el trabajo sea subvalorado o que sus precios queden fuera del mercado. De forma paralela, el sistema funciona como un consultor tecnológico al profesionalizar el producto entregable; utiliza modelos de lenguaje (
LLMs
) que analizan el código fuente para generar automáticamente la documentación técnica y los manuales de usuario. Este paso elimina de raíz la barrera técnica que suele generar desconfianza en las Pymes al momento de operar software desarrollado por universitarios.


2. Antecedentes:


La comercialización de software ha dependido de repositorios que actúan como vitrinas estáticas. En el entorno académico, los proyectos se alojan en sistemas que carecen por completo de un enfoque comercial orientado al usuario final. Por otro lado, los 
marketplaces
 de software actuales operan fundamentalmente como catálogos web tradicionales (sistemas CRUD) que exigen al comprador conocimientos técnicos previos para realizar búsquedas mediante palabras clave rígidas. Además, delegan en el desarrollador la compleja tarea de redactar manuales y fijar precios a ciegas sin orientación del mercado. Este panorama evidencia una carencia significativa: no existe un ecosistema que actúe simultáneamente como asesor comercial para el talento emergente y como consultor tecnológico para el sector empresarial.


Para superar esta limitación, el presente proyecto se fundamenta en la convergencia de paradigmas de la ciencia de datos y la ingeniería de software. Teóricamente, la búsqueda se optimiza mediante Procesamiento del Lenguaje Natural (PLN) y 
embeddings
, permitiendo inferir las necesidades humanas evaluando la similitud semántica de las peticiones. A esto se suma el Machine 
Learning
 Analítico, que emplea algoritmos predictivos para estructurar sistemas de recomendación y optimizar precios con alta fidelidad. Esta inteligencia se apoya en una arquitectura de e-Commerce transaccional que respeta estrictamente las propiedades ACID en bases de datos relacionales, garantizando operaciones financieras seguras. Finalmente, se aplica la teoría de Comprensión de Código (
Code
 
Understanding
) mediante Inteligencia Artificial para traducir lógicas complejas de programación a un lenguaje natural.


Esta conjunción de antecedentes teóricos y tecnológicos se materializa en un caso de uso que redefine la interacción entre la universidad y el sector productivo. Cuando un estudiante sube su proyecto de Taller de Grado, el sistema no solo lo almacena, sino que analiza estadísticamente el software para sugerir un precio óptimo de venta. Simultáneamente, la IA genera de manera automática la documentación técnica y un manual de usuario sencillo. Posteriormente, el dueño de una Pyme ingresa a la plataforma y redacta con sus propias palabras: "Necesito un sistema para mi farmacia que controle inventario y ventas". La inteligencia artificial interpreta esta intención y la conecta directamente con el software del estudiante, logrando un encuentro perfecto entre la oferta universitaria profesionalizada y la demanda del mercado real.




3. Descripción del Problema:

3.1. El fenómeno del "Software de Cajón" en el entorno académico


En facultades como la de Ciencias Computacionales y Telecomunicaciones, la producción de software es constante y de alta calidad. Sin embargo, existe una realidad silenciosa que afecta a casi todos los estudiantes: el proyecto que tomó meses diseñar, programar y testear termina su vida útil el día de la defensa final. Una vez obtenida la nota, ese código —que podría estar gestionando una farmacia, optimizando un inventario o ayudando a un emprendedor— se queda guardado en un disco duro o en un repositorio privado, sin volver a ejecutarse nunca más.


Este escenario representa un desperdicio de talento y recursos. El estudiante invierte cientos de horas en resolver problemas reales, pero al no existir un canal formal para conectar ese resultado con el mundo exterior, el proyecto muere en la academia. El problema no es la falta de calidad del software, sino la falta de un puente que lo lleve al mercado real.

3.2. El desafío de la monetización y la falta de experiencia comercial


Incluso cuando un estudiante tiene la intención de vender su creación, se enfrenta a una pared invisible: no sabe cuánto vale su trabajo. La mayoría de los desarrolladores jóvenes carecen de herramientas para realizar un análisis de mercado o una estructura de costos. Esto deriva en dos extremos peligrosos: o piden precios exageradamente altos que espantan a cualquier comprador, o malbaratan su esfuerzo por montos simbólicos que no recompensan la complejidad técnica del sistema.


A esto se suma la "pesadilla" de la documentación. Para que un software sea comercializable, debe tener manuales de usuario y guías técnicas. Para un programador, redactar estos documentos suele ser una tarea tediosa que consume tiempo que preferiría usar programando. Sin documentación, el software es básicamente "incomprobable" para una persona que no sabe de código, lo que cierra las puertas a cualquier venta formal.

3.3. La Empresa boliviana y la barrera de la transformación digital


Desde la otra vereda, encontramos a las Pequeñas y Medianas Empresas (Pymes) en Santa Cruz y el resto del país. Estos negocios necesitan digitalizarse para sobrevivir, pero se encuentran con presupuestos de desarrollo a medida que son inalcanzables. Una Empresa promedio no puede pagar miles de dólares por un sistema personalizado, pero tampoco sabe dónde buscar soluciones que ya estén hechas y que sean económicas.


El problema aquí es la comunicación. Un dueño de negocio no busca un "
Backend
 en 
FastAPI
 con PostgreSQL"; busca algo que "controle sus ventas y deudas". Los 
marketplaces
 actuales son catálogos fríos que exigen que el comprador sepa términos técnicos para encontrar lo que necesita. Si el comprador no encuentra lo que busca en los primeros cinco minutos o si siente que el sistema es demasiado complejo de instalar, abandona la búsqueda. Existe una brecha de lenguaje y confianza que impide que el software universitario llegue a manos del empresario que lo necesita.

3.4. Análisis del Modelo de Negocio y el Control de Acceso


Para que una solución a este problema sea sostenible, no puede ser simplemente un foro gratuito de descarga de archivos. Se requiere un modelo de negocio que regule el uso y garantice que tanto el comprador como el vendedor perciban valor. El esquema propuesto utiliza un Modelo de Suscripción Freemium, que busca equilibrar el acceso democrático con la rentabilidad de las herramientas avanzadas.


Para el Estudiante (Vendedor): El problema actual es el límite de visibilidad. El plan básico gratuito permite a cualquier estudiante probar la plataforma subiendo un número limitado de proyectos. Sin embargo, un desarrollador prolífico que quiera convertir esto en una carrera de emprendimiento digital necesita subir aplicaciones de forma ilimitada y acceder a herramientas de inteligencia artificial para optimizar sus precios y documentos, lo que justifica un modelo Premium.


Para la Empresa (Comprador): El riesgo de compra es una barrera. El modelo gratuito permite realizar consultas limitadas para explorar soluciones. Si la empresa necesita una búsqueda profunda y constante para equipar diferentes áreas de su negocio, el modelo Premium le otorga acceso ilimitado a la consultoría inteligente de la plataforma.


Este esquema no solo genera ingresos para mantener la infraestructura, sino que actúa como un filtro de calidad y compromiso dentro del ecosistema.

3.5. La ausencia de intermediación inteligente


Finalmente, el problema raíz es que no existe una entidad (humana o tecnológica) que traduzca las necesidades de la Pyme al lenguaje del estudiante y viceversa. Los catálogos actuales son "mudos"; no asesoran al vendedor ni guían al comprador.


Sin un sistema que use Machine 
Learning
 para recomendar la aplicación exacta basándose en el comportamiento del usuario, o que use Procesamiento de Lenguaje Natural para entender un requerimiento sencillo en español, seguiremos viviendo en un ciclo donde los estudiantes "hacen por hacer" y las empresas "necesitan sin encontrar". El problema, en última instancia, es la desconexión total entre el potencial académico de la UAGRM y las necesidades reales del mercado cruceño.


4. Situación Problemática:

El problema central radica en la enorme desconexión que existe entre el software que se crea dentro de la UAGRM y las necesidades reales de las empresas. Semestre a semestre, los estudiantes desarrollan sistemas funcionales que, después de ser calificados, terminan archivados y sin uso comercial porque no existen canales adecuados para ofrecerlos, les falta experiencia para fijar precios y resulta tedioso redactar manuales técnicos. Al mismo tiempo, las pequeñas y medianas empresas (Pymes) necesitan digitalizarse con urgencia, pero se chocan contra costos de desarrollo a medida que no pueden pagar y plataformas que piden demasiados conocimientos técnicos al momento de comprar. Todo esto genera una barrera que perjudica a ambos lados: impide que el estudiante moneticé su trabajo y deja a los negocios sin herramientas accesibles para crecer.


5. Situación deseada:

La implementación de esta plataforma inteligente transforma radicalmente el destino de los proyectos universitarios. El software desarrollado deja de ser una simple tarea que se archiva y pasa a convertirse en un activo digital accesible para cualquier negocio. Al integrar inteligencia artificial, el sistema automatiza las barreras comerciales, permitiendo al estudiante monetizar su esfuerzo de forma continua con precios justos y manuales autogenerados. Del mismo modo, el pequeño empresario, sin necesidad de conocimientos técnicos complejos, encuentra y adquiere exactamente la solución que necesita para optimizar sus ventas o inventarios utilizando un lenguaje cotidiano. En definitiva, se logra construir un puente sólido donde el talento académico impulsa directamente la digitalización y el crecimiento económico del sector productivo.

7. Alcance:

7.1. Módulo de Gestión de Usuarios y Suscripciones (Freemium)


Este módulo será el encargado de controlar quién entra a la plataforma y qué permisos tiene, integrando un ecosistema Freemium.


Gestión de Roles: Creación de perfiles separados para Vendedores (estudiantes/desarrolladores), Compradores (Pymes) y un Administrador general del sistema.


Control de Membresías: Lógica para limitar las acciones según el plan. Para los compradores en plan básico, se limitarán las consultas a la IA a 3 por día; para los vendedores en plan básico, el límite será de 5 aplicaciones en su portafolio. Los usuarios con planes Premium tendrán uso ilimitado de estas funciones.


Pasarela de Pagos (Suscripciones): Integración con la API oficial de 
Stripe
 para delegar la seguridad y el cobro de las membresías.

7.2. Módulo del Comprador y Búsqueda Inteligente (Empresa)


Enfocado en romper la barrera técnica para que cualquier dueño de negocio pueda adquirir software de manera intuitiva.


Motor de Búsqueda Semántica: Un 
prompt
 donde la Pyme escribirá su necesidad utilizando lenguaje natural. Por ejemplo: "Necesito un sistema para mi farmacia que controle inventario y ventas". La IA interpretará esta intención y buscará en la base de datos la aplicación exacta que resuelva el problema.


Sistema de Recomendaciones Personalizadas: Un algoritmo analítico que evaluará el comportamiento y el historial de compras de la Pyme para mostrarle proactivamente la sección "Recomendados para ti".


Asistente Virtual (
Chatbot
 RAG): Un 
bot
 inteligente en la página de cada aplicación que permite al Comprador hacer preguntas técnicas (ej. "¿Este sistema exporta a Excel?"). El 
bot

Biblioteca de Compras: Un espacio privado donde el comprador accederá a sus aplicaciones adquiridas, estadísticas de sus descargas y manuales.

7.3. Módulo del Vendedor y Optimización Comercial (Estudiantes)


Diseñado para que el talento universitario gestione y rentabilice sus proyectos académicos.


Gestión de Portafolio: Interfaces donde el estudiante podrá subir sus proyectos, organizarlos y editar detalles visuales.



Dashboard
 de Rendimiento: Un panel central (como ilustran los prototipos) para que el desarrollador visualice un resumen de su rendimiento, ventas totales, calificación promedio y aplicaciones más vendidas.

Learning

7.4. Módulo de IA para Comprensión de Código y Documentación


Este es el puente técnico que profesionaliza el producto académico y elimina la barrera técnica con el comprador.


Lectura de Código (
Code
 
Understanding

LLMs
) para analizar profundamente la estructura del código subido por el estudiante.


Generador de Documentación Técnica: Creación automática de diagramas de flujo lógicos, descripción de módulos y requisitos del sistema.


Generador de Manual de Usuario: Redacción automática de una guía paso a paso, utilizando un lenguaje sencillo, para que el comprador sepa exactamente cómo instalar y operar el sistema.

El motor de e-Commerce transaccional que asegura un intercambio financiero y de datos puro.

TypeScript

Como estamos usando microservicios, no meteremos todo en una sola base de datos, sino que usaremos la mejor herramienta para cada tarea:



  PostgreSQL
: Es nuestro motor relacional principal y es extremadamente robusto. Aquí guardaremos lo más delicado: las cuentas de usuario, las contraseñas, los cobros y los historiales de compras. Su trabajo es garantizar que las transacciones financieras sean puras y no permitan errores.



  Extensión
 Vectorial (
pgvector
): En lugar de instalar una base de datos separada para la inteligencia artificial, instalaremos esta extensión directamente dentro de PostgreSQL. Aquí guardaremos los "vectores matemáticos" que crea la IA al leer el código de los estudiantes. Gracias a esto, el buscador semántico y el Asistente Virtual (RAG) podrán entender lo que escribe el comprador y conectarlo con la aplicación perfecta.


8.7. Inteligencia Artificial y Análisis de Datos



OpenAI
 API: Será el "cerebro" prestado de nuestro sistema. Lo usaremos para leer el código de los estudiantes, generar los manuales automáticamente y entender las búsquedas en lenguaje natural.


Pandas y 
Scikit-learn
: Son herramientas matemáticas de Python. Las usaremos para analizar el comportamiento del mercado y sugerirle al estudiante qué precio debería ponerle a su software.


8.8. Despliegue y Servicios Externos



Stripe
: Es la pasarela de pagos oficial que integraremos. En lugar de programar un sistema de tarjetas de crédito desde cero (lo cual es muy peligroso), delegamos toda esa seguridad a 
Stripe
 para que cobre las suscripciones de forma confiable.

1. Filtro del Desarrollador: Pruebas Unitarias (Lógica de Código)


Estas pruebas garantizan que las funciones individuales escritas en React.js y 
FastAPI
 operen sin errores.


Prueba de Encriptación (
Backend
): Se validó que la función encargada de registrar usuarios aplique correctamente un algoritmo de 
hashing
 a la contraseña antes de insertarla en PostgreSQL. No es posible guardar contraseñas en texto plano bajo ninguna circunstancia.


Prueba de Generación de Token (
Backend
): Se verificó 
que
 al procesar un inicio de sesión exitoso, el sistema genere un JWT (JSON Web Token) válido, que contenga el ID del usuario y su Rol (VENDEDOR o COMPRADOR), con un tiempo de expiración definido.


Prueba de Validación de Formularios (
Frontend
): Se probó la interfaz de 
React
 para asegurar que el botón de "Registrar" se desactive si el correo electrónico no tiene un formato válido (ausencia del "@") o si la contraseña tiene menos de 8 caracteres, evitando enviar peticiones basura al servidor.

2. Filtro de QA: Pruebas de Integración y Rendimiento (Tiempos de Respuesta)


El objetivo de estas pruebas fue confirmar que el 
Frontend
 y el 
Backend
 se comuniquen fluidamente y soporten la carga de usuarios sin lentitud.


Integración 
End-to-End
 (Registro y 
Login
): Se simuló el flujo completo: creación de un usuario desde la web, recepción de los datos en la API REST (
FastAPI
) y la correcta creación del registro en la tabla usuarios de PostgreSQL. Inmediatamente después, se probó el inicio de sesión con esas mismas credenciales, confirmando un código de estado HTTP 200 (OK).


Prueba de Rendimiento (Tiempo de Respuesta): Se realizaron múltiples peticiones de inicio de sesión simultáneas. El servidor (
FastAPI
) demostró la capacidad de procesar la autenticación y devolver el token en un tiempo promedio inferior a 300 milisegundos, cumpliendo con el requerimiento de alta velocidad esperado para una PWA.


Prueba de Tolerancia a Errores: Se forzó el inicio de sesión con credenciales incorrectas para asegurar que el sistema devuelva de forma rápida y segura un error HTTP 401 (
Unauthorized
), mostrando el mensaje correcto en la interfaz sin exponer detalles de la base de datos.

3. Filtro del 
Product
 
Owner
: Pruebas de Aceptación (Requerimientos de Negocio)

Qué problemas se encontró y cómo
